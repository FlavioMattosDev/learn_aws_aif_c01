Domínio 1: Fundamentos de IA e ML

20% do exame — Conceitos fundamentais de Machine Learning

📊 Tipos de Aprendizado de Máquina

Supervisionado Mais Comum

Treina com dados rotulados (entrada → saída conhecida). Usado para classificação e regressão.

  • Classificação: prever categorias (spam/não spam, fraude/legítimo)
  • Regressão: prever valores contínuos (preço, temperatura, tempo de entrega)

Não Supervisionado

Treina com dados sem rótulos — descobre padrões e estruturas ocultas nos dados.

  • Clustering: agrupa dados similares (K-Means, DBSCAN)
  • Redução de dimensionalidade: PCA, t-SNE
  • Detecção de anomalias: identifica outliers

Por Reforço

Um agente aprende por tentativa e erro, recebendo recompensas ou penalidades por suas ações.

  • Robótica e controle autônomo
  • Jogos e otimização de estratégias
  • Sistemas de recomendação adaptáveis
💡 Dica para o Exame

Se o cenário tem dados rotulados → Supervisionado. Se não tem rótulos e quer descobrir padrões → Não supervisionado. Se aprende por interação e feedback contínuo → Reforço.

🎯 Regressão vs Classificação

Aspecto Regressão Classificação
Saída Valor numérico contínuo Categoria discreta
Exemplos Preço de imóvel, tempo de entrega, receita prevista Spam/não spam, doente/saudável, tipo de produto
Algoritmos Regressão Linear, Random Forest Regressor, XGBoost Logistic Regression, SVM, Decision Trees, Neural Networks
Métricas MSE, RMSE, MAE, R² Acurácia, Precisão, Recall, F1, AUC-ROC

📈 Métricas de Avaliação

Precisão (Precision)

Dos que o modelo classificou como positivo, quantos realmente são positivos?

Use quando: falsos positivos são caros (ex: bloquear e-mail legítimo)

Recall (Sensibilidade)

De todos os positivos reais, quantos o modelo conseguiu detectar?

Use quando: falsos negativos são perigosos (ex: não detectar câncer)

F1-Score

Média harmônica entre Precisão e Recall. Equilibra ambos quando não há prioridade clara.

Use quando: precisão e recall são igualmente importantes

AUC-ROC

Área sob a curva ROC — avalia a capacidade do modelo de distinguir classes em todos os limiares.

Use quando: quer comparar modelos em múltiplos pontos de corte

⚠️ Cuidado com Acurácia

Acurácia pode ser enganosa com dados desbalanceados. Um modelo que sempre diz "não fraude" em um dataset com 99% de transações legítimas terá 99% de acurácia mas será inútil para detectar fraudes.

⚖️ Overfitting vs Underfitting

Aspecto Overfitting Underfitting
O que é Modelo memorizou os dados de treino Modelo não aprendeu padrões suficientes
Sintoma Alta performance no treino, baixa no teste Baixa performance em ambos
Soluções Mais dados, regularização, dropout, cross-validation, reduzir complexidade Modelo mais complexo, mais features, mais épocas de treino

🔧 Preparação de Dados e Feature Engineering

  • Valores ausentes: Imputação (média, mediana, moda) ou remoção de registros
  • Dados desbalanceados: Oversampling (SMOTE), undersampling, ajuste de pesos de classe
  • Normalização/Padronização: Escalar features para mesma faixa (Min-Max, Z-score)
  • Encoding: Transformar variáveis categóricas em numéricas (One-Hot, Label Encoding)
  • Feature Engineering: Criar novas variáveis a partir das existentes para melhorar o modelo
  • Divisão de dados: Treino (70-80%) → Validação (10-15%) → Teste (10-15%)
✅ Papel de cada conjunto

Treino: o modelo aprende padrões. Validação: ajustar hiperparâmetros e monitorar overfitting. Teste: avaliação final em dados totalmente inéditos.

Domínio 2: Fundamentos de IA Generativa

24% do exame — LLMs, Foundation Models, Amazon Bedrock

🏗️ Modelos de Fundação (Foundation Models)

Modelos de fundação são modelos de IA treinados em grandes volumes de dados que podem ser adaptados para diversas tarefas. São a base da IA generativa moderna.

Características

  • Pré-treinados em datasets massivos
  • Generalistas — adaptáveis a muitas tarefas
  • Bilhões de parâmetros
  • Aprendem representações profundas da linguagem

LLMs (Large Language Models)

  • Tipo específico de foundation model para texto
  • Geram, resumem, traduzem e analisam texto
  • Exemplos: Claude (Anthropic), Llama (Meta), Titan (Amazon)
  • Base dos chatbots e assistentes modernos

Modelos Multimodais

  • Processam múltiplos tipos de entrada (texto, imagem, áudio)
  • Geram diferentes tipos de saída
  • Exemplos: geração de imagens a partir de texto
  • Análise de documentos com texto e imagens

🔄 Arquitetura Transformer (Conceitos Básicos)

A arquitetura Transformer é a base dos LLMs modernos. Conceitos-chave para o exame:

  • Self-Attention (Atenção): permite ao modelo ponderar a importância de cada palavra em relação às outras na sequência
  • Tokens: unidades de texto processadas (palavras ou subpalavras). O modelo trabalha com tokens, não palavras inteiras
  • Embeddings: representações numéricas (vetores) que capturam o significado semântico do texto
  • Paralelização: diferente de RNNs, Transformers processam todos os tokens em paralelo → muito mais rápido
🔑 Conceito-chave: Embeddings

Embeddings transformam texto em vetores numéricos onde textos com significado similar ficam próximos no espaço vetorial. São a base da busca semântica e do RAG.

⚙️ Treinamento vs Inferência

Aspecto Treinamento Inferência
O que é O modelo aprende padrões dos dados O modelo usa o que aprendeu para gerar respostas
Quando ocorre Antes do deploy (fase de desenvolvimento) Em produção, a cada requisição do usuário
Custo Muito alto (GPUs por dias/semanas) Menor (por token gerado)
Resultado Pesos do modelo ajustados Texto/imagem/áudio gerado

🛠️ Fine-tuning vs RAG vs Prompt Engineering

Abordagem O que faz Quando usar Custo
Prompt Engineering Ajusta a entrada (prompt) para guiar o modelo Controle rápido sem modificar o modelo 💚 Baixo
RAG Recupera informações externas e as fornece ao modelo junto com a pergunta Dados que mudam frequentemente; precisão factual 💛 Médio
Fine-tuning Retreina o modelo com dados específicos, alterando seus pesos Adaptar tom, estilo ou domínio especializado 🔴 Alto
⚠️ Pergunta frequente no exame

Se a base de conhecimento muda frequentemente → RAG. Se precisa adaptar estilo/tom permanentemente → Fine-tuning. Se quer resultado rápido sem custo → Prompt Engineering.

🎚️ Parâmetros de Inferência

Temperature Essencial

Controla a aleatoriedade das respostas.

  • Baixa (0-0.3): determinística, factual
  • Alta (0.7-1.0): criativa, variada

Top-P (Nucleus Sampling)

Limita as opções de tokens ao top-P% mais prováveis. Complementa a temperatura.

  • Baixo (0.1-0.3): respostas focadas
  • Alto (0.9-1.0): mais diversidade

Max Tokens

Limite máximo de tokens na resposta gerada. Se for muito baixo, a resposta pode ser cortada no meio.

Stop Sequences

Sequências de texto que, quando geradas, fazem o modelo parar. Útil para controlar formato de saída.

💡 Para o exame

Resposta cortada no meio → aumente max tokens. Resposta repetitiva/sem criatividade → aumente temperature. Quer mais controle sobre o vocabulário → ajuste top-p.

📐 Conceitos Importantes

Context Window (Janela de Contexto)

Número máximo de tokens que o modelo pode processar de uma vez (entrada + saída). Modelo "esquece" o início em conversas longas quando ultrapassa esse limite.

Alucinações (Hallucinations)

O modelo gera informações fluentes mas factualmente incorretas. Parece confiante, mas está inventando. RAG e Guardrails ajudam a mitigar.

Tokens

Unidade de processamento do LLM. ~1 token ≈ ¾ de uma palavra em inglês. O custo é calculado por token (entrada + saída).

🟧 Amazon Bedrock

Serviço gerenciado da AWS que dá acesso a múltiplos modelos de fundação via API, sem gerenciar infraestrutura.

  • Multi-modelo: Claude (Anthropic), Llama (Meta), Titan (Amazon), Mistral, Stability AI, Cohere
  • Sem infraestrutura: pague por uso (tokens processados), sem provisionar GPUs
  • Privacidade: dados de clientes NÃO são usados para treinar modelos base
  • Customização: fine-tuning e continued pre-training nos seus dados
  • Avaliação de modelos: compare modelos para sua tarefa específica
✅ Critérios de seleção de modelo no Bedrock

Considere: janela de contexto (documentos longos), velocidade de inferência (latência), custo por token, qualidade para a tarefa (sumarização, código, conversação), e idiomas suportados.

Domínio 3: Aplicações de Modelos de Base

28% do exame — O domínio mais importante! RAG, Agents, Prompt Engineering

🏛️ Design Patterns de IA Generativa

RAG (Retrieval-Augmented Generation) Muito Cobrado

Recupera informações relevantes de uma base de dados externa e as fornece ao modelo junto com a pergunta. Reduz alucinações e mantém respostas atualizadas.

  • 1. Pergunta do usuário → embedding
  • 2. Busca vetorial na base de conhecimento
  • 3. Documentos relevantes recuperados
  • 4. Contexto + pergunta enviados ao LLM
  • 5. Resposta ancorada em fatos reais

Agents (Agentes de IA)

Sistemas que usam LLMs para planejar e executar tarefas complexas de múltiplos passos, invocando ferramentas externas de forma autônoma.

  • Planejam sequências de ações
  • Usam APIs, bancos de dados, buscas
  • Tomam decisões intermediárias
  • Executam ações no mundo real

Chains (Encadeamentos)

Sequências de chamadas a LLMs onde a saída de uma etapa alimenta a entrada da próxima. Útil para tarefas complexas divididas em passos.

  • Decomposição de problemas complexos
  • Pipeline de processamento de texto
  • Validação e refinamento iterativo

🟧 Amazon Bedrock — Features Avançadas

Knowledge Bases RAG Gerenciado

Implementação gerenciada de RAG. Conecta seus dados (S3, web crawlers) ao Bedrock automaticamente. Gerencia embeddings, indexação e busca vetorial.

Agents for Bedrock

Cria agentes que planejam e executam tarefas usando APIs e knowledge bases. Orquestra múltiplas ações sem código de orquestração manual.

Guardrails

Filtros de segurança configuráveis: bloqueia tópicos sensíveis, filtra informações pessoais (PII), impede respostas inadequadas. Defesa contra prompt injection.

Model Evaluation

Compare modelos automaticamente para sua tarefa. Métricas de qualidade, latência e custo para tomar decisões informadas de seleção de modelo.

💡 Knowledge Bases vs RAG manual

Knowledge Bases automatiza todo o pipeline RAG (chunking, embedding, indexação, retrieval). RAG manual exige configurar cada etapa, mas oferece mais controle.

📝 Técnicas de Prompt Engineering

Técnica Descrição Quando Usar
Zero-shot Instruir o modelo sem fornecer exemplos Tarefas simples e gerais; quando o modelo já sabe fazer
Few-shot Fornecer 2-5 exemplos do formato/comportamento desejado Garantir formato específico de saída; tarefas especializadas
Chain-of-Thought (CoT) Pedir ao modelo para "pensar passo a passo" Raciocínio lógico, matemática, problemas complexos
System Prompt Definir regras, persona e restrições do modelo Controlar comportamento sem modificar o modelo
✅ Boas práticas de prompt

Seja específico e claro. Defina o formato de saída. Forneça contexto relevante. Use delimitadores para separar instruções dos dados. Peça ao modelo para dizer "não sei" quando não tiver certeza.

✂️ Estratégias de Chunking

Para RAG funcionar bem, documentos precisam ser divididos em pedaços (chunks) adequados:

Fixed-size Chunking

Divide em pedaços de tamanho fixo (ex: 512 tokens). Simples mas pode cortar informação no meio.

Semantic Chunking

Divide por significado — respeita parágrafos, seções e limites lógicos do documento.

Overlap (Sobreposição)

Chunks compartilham parte do texto com o anterior/próximo para manter contexto nas bordas.

💻 Amazon Q Developer (ex-CodeWhisperer)

Assistente de IA para desenvolvedores integrado às IDEs:

  • Autocompletar código: sugestões em tempo real enquanto digita
  • Geração de código: cria funções e classes a partir de comentários em linguagem natural
  • Scan de segurança: identifica vulnerabilidades e sugere correções
  • Explicação de código: explica código existente
  • Transformação de código: atualiza versões de linguagens/frameworks

📦 Batch Inference vs Real-time Inference

Aspecto Real-time (On-demand) Batch
Uso Chatbots, assistentes, respostas imediatas Processar milhares de documentos de uma vez
Latência Milissegundos a segundos Minutos a horas (não importa)
Custo Maior por token individual Menor por token (economia de escala)

Domínio 4: Diretrizes de IA Responsável

14% do exame — Ética, equidade, transparência e governança

🎯 Princípios de IA Responsável

Equidade (Fairness)

O modelo deve tratar todos os grupos de forma justa e não discriminar com base em raça, gênero, idade, renda etc. Desempenho deve ser equitativo entre subgrupos.

Explicabilidade (Explainability)

Deve ser possível entender por que o modelo tomou determinada decisão. Importante para decisões de alto impacto (crédito, saúde, justiça).

Transparência

O sistema deve ser honesto sobre sua natureza — identificar-se como IA quando questionado. Documentar limitações e capacidades.

Robustez e Segurança

O modelo deve funcionar de forma confiável, resistir a ataques adversariais e falhar de forma segura (graceful degradation).

Privacidade

Proteger dados pessoais usados no treinamento e na inferência. Não vazar informações sensíveis nas respostas.

Governança

Processos organizacionais para supervisão, auditoria, responsabilização e melhoria contínua dos sistemas de IA.

🔍 Viés (Bias) em IA

Tipo de Viés Descrição Exemplo
Viés de dados (Data Bias) Dados de treinamento não representam a população igualmente Modelo treinado majoritariamente com dados de uma região falha em outras
Viés de seleção Amostra de treinamento não é representativa Modelo de crédito treinado apenas com clientes existentes
Viés algorítmico O próprio algoritmo amplifica desigualdades Otimizar para acurácia geral mascara falhas em minorias
🚨 Caso clássico do exame

Modelo com 95% de acurácia geral, mas desempenho muito inferior para um grupo minoritário = problema de equidade (fairness), não de acurácia. A solução envolve dados mais representativos e métricas por subgrupo.

👤 Supervisão Humana (Human-in-the-Loop)

Em decisões de alto impacto (saúde, crédito, justiça), humanos devem permanecer no processo decisório:

  • Human-in-the-loop: humano revisa cada decisão antes de executá-la
  • Human-on-the-loop: humano monitora e pode intervir, mas a IA age por padrão
  • Human-over-the-loop: humano define regras e políticas, IA opera dentro desses limites
💡 Para o exame

"IA centrada no ser humano" (Human-centered AI) = manter supervisão humana em decisões de alto impacto. Nunca substituir completamente humanos em decisões que afetam vidas.

🛡️ Ferramentas AWS para IA Responsável

Amazon Bedrock Guardrails

Filtros configuráveis para bloquear conteúdo tóxico, informações pessoais (PII) e tópicos sensíveis nas respostas da IA.

Domínio 5: Segurança, Conformidade e Governança

14% do exame — Proteção de dados, IAM, compliance e monitoramento

🔐 Segurança de Dados para IA

Criptografia em Trânsito

TLS/SSL para todas as comunicações com serviços AWS de IA. Dados nunca trafegam em texto puro.

Criptografia em Repouso

AWS KMS para criptografar dados armazenados (modelos, datasets, embeddings). Chaves gerenciadas pelo cliente (CMK) para máximo controle.

Privacidade no Bedrock

Por padrão, dados de clientes NÃO são usados para treinar ou melhorar modelos base da AWS. Isolamento total.

VPC Endpoints (PrivateLink)

Conecta sua VPC ao Bedrock pela rede privada da AWS. Dados nunca passam pela internet pública.

🎫 IAM para Serviços de IA

O AWS IAM controla quem pode fazer o quê com os serviços de IA:

  • Princípio do menor privilégio: dar apenas as permissões necessárias para a tarefa
  • Policies granulares: controlar quais modelos cada equipe pode invocar no Bedrock
  • IAM Roles: para aplicações que chamam Bedrock (não usar access keys hardcoded)
  • Service Control Policies (SCPs): limites em nível de organização (AWS Organizations)
  • Resource-based policies: controlar acesso a modelos customizados específicos
⚠️ Não confunda

IAM: controle granular de acesso (quem pode invocar qual modelo). Amazon Cognito: identidade de usuários finais de apps. Security Groups: controle de rede, não de permissões de serviço.

🛡️ Ameaças e Proteções para IA

Prompt Injection Importante

Atacante insere instruções maliciosas no input para manipular o LLM a ignorar suas regras de segurança.

Defesa: Bedrock Guardrails, validação de input, system prompts robustos

Data Poisoning

Atacante contamina os dados de treinamento para degradar o modelo ou inserir backdoors.

Defesa: Validação de dados, monitoramento de qualidade, data lineage

Model Extraction

Atacante faz muitas queries para recriar o modelo ou extrair dados de treinamento.

Defesa: Rate limiting, monitoramento de uso anormal, watermarking

Vazamento de Dados Sensíveis

Modelo gera informações confidenciais que estavam no contexto ou nos dados de treino.

Defesa: Guardrails com filtro de PII, revisão de prompts, sanitização

📊 Monitoramento e Auditoria

AWS CloudTrail

Registra TODAS as chamadas de API (quem invocou, quando, qual modelo, qual IP). Essencial para auditoria.

Amazon CloudWatch

Métricas operacionais: latência, tokens processados, erros, throughput. Alarmes e dashboards.

AWS Config

Rastreia mudanças em configurações de recursos. Compliance rules para verificar configurações de segurança.

Amazon GuardDuty

Detecção de ameaças: identifica comportamentos maliciosos e acessos não autorizados usando ML.
